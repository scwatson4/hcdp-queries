# Query: I do not see security groups in Exosphere — do I need to shelve the instance?

**Date:** 2026-04-30 09:15 UTC

---

No, you don't shelve. Exosphere is a simplified frontend and doesn't expose security group editing in its UI. Two options:

## Option 1 (preferred): Use Horizon

Horizon is the full OpenStack web UI for Jetstream2 and lets you manage security groups directly.

1. Go to **https://js2.jetstream-cloud.org**
2. Log in with the same Globus / ACCESS credentials you use for Exosphere
3. Top nav: **Project** → **Network** → **Security Groups**
4. Find the security group attached to your instance (Exosphere usually creates one named `exosphere` or similar; `default` is also common)
5. Click **Manage Rules** → **Add Rule**
6. Set:
   - **Rule:** Custom TCP Rule
   - **Direction:** Ingress
   - **Open Port:** Port (not Port Range)
   - **Port:** `8000`
   - **CIDR:** `0.0.0.0/0` (open to internet — easiest) or your chatbot's IP `/32` (locked down)
7. Click **Add**

Effect is instant — no instance restart needed.

## Option 2: openstack CLI

If you have OpenStack CLI credentials (an `openrc.sh` file from Exosphere's "Credentials" page or Horizon's API Access page):

```bash
# From your laptop after sourcing openrc.sh
openstack security group rule create --proto tcp --dst-port 8000 --remote-ip 0.0.0.0/0 <security-group-name>
```

## How to verify it worked

From any machine outside Jetstream2:

```bash
curl http://149.165.155.217:8000/health
```

Should return `{"status":"ok",...}`. If it hangs, the firewall rule isn't applied yet. If it returns immediately, you're done.

## Why Horizon and not Exosphere?

Exosphere prioritizes ease of use for common scientific computing tasks (instance creation, volume management, basic SSH). Network-level configuration like security groups is left to Horizon, which exposes the full OpenStack feature set. Both manage the same underlying project — anything you change in Horizon is reflected in Exosphere immediately.
